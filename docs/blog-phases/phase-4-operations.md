# Phase 4 - Operations

## 목표

이 phase의 목적은 블로그를 단순히 동작하는 기능 집합이 아니라 운영 가능한 제품으로 마무리하는 것이다. 검색엔진 대응, 링크 공유, 빈 상태와 오류 상태, 재검증 흐름을 정리해 실제 서비스 품질을 끌어올린다.

핵심 목표는 다음과 같다.

- metadata와 공유 미리보기 기반을 정리한다.
- sitemap, robots 같은 검색엔진 대응 요소를 추가한다.
- 빈 상태, 오류 상태, not-found 흐름을 정리한다.
- 발행/수정 후 공개 페이지 반영 정책을 정리한다.

## 범위

포함 항목은 다음과 같다.

- 목록/상세/시리즈 metadata 정리
- sitemap과 robots
- canonical URL 정책
- empty state, error state, loading state 정리
- publish/update/delete 이후 공개 페이지 반영 전략

제외 항목은 다음과 같다.

- RSS
- 검색
- 태그 SEO 확장
- 분석 대시보드
- 다국어 대응

## 메타데이터 정책

다음 페이지들은 모두 metadata를 가져야 한다.

- `/blog`
- `/memo`
- `/booklog`
- `/:type/[slug]`
- `/booklog/series/[seriesSlug]`

정책은 다음처럼 고정한다.

- 목록 페이지는 카테고리 성격이 드러나는 title/description을 가진다.
- 포스트 상세는 포스트 제목과 요약을 metadata에 반영한다.
- BOOKLOG 시리즈 상세는 시리즈 제목과 설명을 metadata에 반영한다.
- canonical URL은 현재 라우트 구조 기준으로 고정한다.

이 phase에서는 동적 OG 이미지 생성까지는 필수가 아니다. 우선은 정적 기본 이미지 또는 공용 기본값으로 충분하다.

## sitemap과 robots

sitemap에는 다음 항목을 포함한다.

- 홈 또는 블로그 진입 페이지
- `/blog`, `/memo`, `/booklog`
- 모든 published post 상세
- 모든 공개 BOOKLOG 시리즈 상세

robots는 기본 크롤링 허용을 전제로 한다. 다만 작성 라우트와 인증 라우트는 색인 대상이 아니어야 한다.

## 페이지 상태 정리

다음 상태를 의도적으로 설계한다.

- 포스트가 없을 때의 목록 empty state
- 시리즈가 비어 있을 때의 empty state
- 존재하지 않는 포스트/시리즈의 not-found
- 서버 오류가 발생했을 때의 error state
- 느린 응답 시의 loading state

원칙은 다음과 같다.

- empty state는 단순 "없음"이 아니라 해당 영역의 의미를 다시 설명해야 한다.
- not-found는 URL이 잘못되었는지, 삭제되었는지 사용자가 이해할 수 있어야 한다.
- loading UI는 모바일에서도 레이아웃 점프가 크지 않아야 한다.

## 공개 반영 전략

발행/수정/삭제 이후 공개 페이지 반영 정책을 이 phase에서 명시적으로 고정한다.

- owner가 publish 또는 update를 완료하면 관련 목록과 상세 라우트가 갱신되어야 한다.
- 댓글 삭제나 좋아요 증감은 상호작용 영역 단위 갱신으로 충분하다.
- 읽기 페이지 캐싱은 공격적으로 최적화하기보다, 정확성을 우선하는 보수적 전략으로 시작한다.

이 문서 기준의 기본 전략은 다음과 같다.

- 공개 목록과 상세는 서버 기준 읽기 흐름을 유지한다.
- owner mutation 후에는 관련 페이지 재검증을 수행한다.
- 어떤 캐시 키를 쓸지보다 어떤 사용자 경험을 보장할지가 우선이다.

## 운영 체크리스트

배포 전 최소 점검 항목은 다음과 같다.

- published post만 sitemap에 포함되는지
- draft가 metadata나 공개 경로에 노출되지 않는지
- not-found와 error UI가 깨지지 않는지
- 공유 링크 제목과 설명이 의도대로 보이는지
- mobile viewport에서 empty/loading/error 상태가 읽기 쉬운지

## 검증 항목

- 주요 목록과 상세 페이지의 metadata가 올바른지 확인한다.
- sitemap에 공개 리소스만 포함되는지 확인한다.
- 작성 페이지나 인증 라우트가 색인 대상이 아닌지 확인한다.
- publish 직후 상세와 목록 페이지가 갱신되는지 확인한다.
- 빈 카테고리와 잘못된 slug 접근에 대한 상태 표현이 적절한지 확인한다.

## 완료 조건

- 블로그가 검색엔진과 링크 공유 관점에서 기본적인 제품 품질을 갖춘다.
- 운영자가 발행 후 공개 반영 여부를 예측할 수 있다.
- 빈 상태, 오류 상태, not-found가 의도적으로 설계되어 있다.
- 이후 추가 기능을 붙여도 공개 라우트의 운영 품질 기준이 유지된다.

## 이번 phase에서 하지 않는 것

- 분석 리포트
- 소셜 카드 자동 생성 고도화
- 알림과 운영 대시보드
