# Blog Plan

## 목적

이 문서는 이 프로젝트에 블로그 기능을 도입하기 위한 1차 기준 문서다.

- 구현보다 문서화를 먼저 진행한다.
- 상위 기준서는 이 문서 하나로 유지한다.
- 세부 구현 이슈, 작업 분할, 일정 관리는 이후 이 문서를 기준으로 별도 정리한다.

Phase별 상세 구현 계획은 다음 하위 문서에서 관리한다.

- [Blog Phase Docs](./blog-phases/README.md)

이 문서는 `docs/architecture.md`의 계층 규칙을 전제로 한다. 따라서 아키텍처 일반론을 반복하지 않고, 블로그 기능이 현재 프로젝트 구조에 어떻게 매핑되는지만 정리한다.

## 개요

이 블로그는 단순한 게시판이 아니라 기록의 성격이 다른 세 가지 콘텐츠를 한 제품 안에서 운영하기 위한 공간이다.

- `BLOG`: 주제를 갖추고 정보를 전달하는 장문 포스트
- `MEMO`: 개발 중 발견한 에러, 짧은 학습 기록, 메모성 포스트
- `BOOKLOG`: 책을 읽으며 정리한 기록, 챕터 요약, 시리즈형 포스트

세 타입은 같은 블로그 제품 안에 속하지만, 독자에게 기대하는 읽기 경험이 다르다.

- `BLOG`는 완결된 글을 읽는 경험에 집중한다.
- `MEMO`는 빠르게 훑고 참고하는 경험에 집중한다.
- `BOOKLOG`는 시리즈 흐름과 순서 탐색에 집중한다.

## 핵심 원칙

- UI는 모바일 반응형을 먼저 설계하고, 같은 구조를 데스크탑으로 확장한다.
- 라이트 모드와 다크 모드를 모두 지원한다.
- 권한은 비로그인 사용자, 로그인 사용자, 주인(owner)으로 명확히 분리한다.
- 로그인한 일반 사용자는 상호작용만 할 수 있고, 글 작성 권한은 없다.
- 글 작성과 수정은 상세 페이지 인라인 편집이 아니라 전용 작성 페이지에서 처리한다.
- 발행 워크플로우는 `초안 + 발행` 2단계로 운영한다.
- 댓글은 즉시 공개하며 승인 단계는 넣지 않는다.
- 에디터 1차 범위는 텍스트 중심으로 제한하고, 본문 이미지 업로드는 후속 단계로 미룬다.

## 역할과 권한

주인은 단일 GitHub 계정 1개를 기준으로 식별한다. 구현 단계에서는 GitHub provider 계정 식별자 기반 판별을 기본값으로 잡는다.

| 역할 | 로그인 여부 | 읽기 | 좋아요 | 댓글 | 대댓글 | 글 작성 | 글 수정 | 발행 | 댓글 삭제 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 비로그인 사용자 | 아니오 | 가능 | 불가 | 불가 | 불가 | 불가 | 불가 | 불가 | 불가 |
| 로그인 사용자 | 예 | 가능 | 가능 | 가능 | 가능 | 불가 | 불가 | 불가 | 불가 |
| 주인(owner) | 예 | 가능 | 가능 | 가능 | 가능 | 가능 | 가능 | 가능 | 가능 |

권한 정책은 다음 원칙으로 고정한다.

- 좋아요는 로그인 사용자 이상만 가능하다.
- 댓글과 대댓글은 로그인 사용자 이상만 가능하다.
- 대댓글은 1단계까지만 허용한다.
- 포스트 작성, 수정, 초안 저장, 발행은 owner만 가능하다.
- 댓글 삭제는 owner만 가능하다.

## 정보 구조와 라우팅

### 라우팅 원칙

- 공개 읽기 라우트와 작성 라우트를 분리한다.
- 사용자가 인지하는 URL은 카테고리 단위로 명확하게 나눈다.
- 문서상 `:type`은 개념적 패턴이며 실제 구현에서는 `blog`, `memo`, `booklog` 각 세그먼트로 풀어낸다.

### 최소 라우트

| 경로 | 목적 | 주요 섹션 | 모바일 우선 원칙 |
| --- | --- | --- | --- |
| `/blog` | BLOG 목록 | 상단 소개, 필터 또는 정렬, 포스트 리스트 | 카드 간격과 타이포는 좁은 화면에서도 빠르게 훑을 수 있게 유지 |
| `/memo` | MEMO 목록 | 짧은 포스트 리스트, 날짜/태그성 메타 | 한 화면에 더 많은 기록을 볼 수 있도록 정보 밀도를 다소 높게 유지 |
| `/booklog` | BOOKLOG 목록 | 시리즈 리스트, 최신 BOOKLOG 포스트 | 시리즈 탐색을 우선하고, 카드보다 순서 정보가 먼저 보이게 구성 |
| `/:type/[slug]` | 포스트 상세 | 제목, 메타, 본문, 좋아요, 댓글, 대댓글 | 본문 가독성, 액션 영역 분리, 하단 상호작용 고정성 고려 |
| `/booklog/series/[seriesSlug]` | BOOKLOG 시리즈 상세 | 시리즈 소개, 챕터/파트 목록, 관련 포스트 순서 | 작은 화면에서도 순서를 잃지 않도록 리스트 구조를 우선 |
| `/write` | owner 작성 진입 | 카테고리 선택, 초안 목록, 빠른 작성 진입 | 관리 기능을 복잡한 대시보드보다 단순한 진입 구조로 유지 |
| `/:type/new` | 새 글 작성 | 제목, 메타, Lexical 에디터, 저장/발행 액션 | 작성 화면은 한 컬럼 중심으로 유지하고 보조 액션은 접을 수 있게 구성 |
| `/:type/[slug]/edit` | 기존 글 수정 | 기존 콘텐츠 로드, 상태 표시, 수정/저장/발행 액션 | 편집 기능과 미리보기를 한 화면에 과도하게 겹치지 않도록 유지 |

### 페이지 성격

- `BLOG` 상세는 정보 전달형 장문 읽기에 맞는 넓은 본문 영역을 기본으로 한다.
- `MEMO` 상세는 빠르게 읽고 참고할 수 있게 메타 정보와 본문 밀도를 높인다.
- `BOOKLOG` 상세는 단일 글 읽기뿐 아니라 시리즈 안에서의 위치가 함께 드러나야 한다.

## 콘텐츠 모델

### 핵심 상태

- `PostType = BLOG | MEMO | BOOKLOG`
- `PostStatus = DRAFT | PUBLISHED`

### 포스트 모델 원칙

모든 포스트는 기본적으로 하나의 공통 모델을 사용한다. `BOOKLOG`만 별도 도메인으로 분리하지 않고, 공통 post 모델 위에 추가 메타를 얹는 방식으로 운영한다.

- 공통 필드 예시: 제목, 슬러그, 타입, 상태, 본문 원본, 렌더 결과, 발행 시각, 작성자
- `BOOKLOG` 추가 메타 예시: `series`, `order`, `chapterLabel`

이 결정의 목적은 다음과 같다.

- 세 콘텐츠 타입의 공통 작성 흐름을 유지한다.
- 목록, 상세, 좋아요, 댓글 시스템을 공유한다.
- `BOOKLOG`만 순서 정보와 시리즈 정보를 추가로 표현할 수 있게 한다.

### 댓글과 좋아요 모델 원칙

- 좋아요는 `사용자-포스트` 단위의 toggle 구조로 본다.
- 한 사용자는 같은 포스트에 한 번만 좋아요 상태를 가질 수 있다.
- 댓글은 포스트 단위로 연결한다.
- 댓글의 depth는 `0` 또는 `1`만 허용한다.
- 대댓글의 대댓글은 허용하지 않는다.
- 댓글 삭제 권한은 owner에게만 있다.

## 기술 설계

### 기술 스택 고정

- 프런트엔드: Next.js 16 App Router
- 클라이언트 데이터 계층: TanStack Query + tRPC
- 에디터: Lexical
- 인증: GitHub OAuth with Auth.js
- 데이터베이스: Neon
- ORM: Drizzle
- 프런트엔드-백엔드 통신: tRPC

### 저장 기준

에디터 데이터는 다음 흐름을 기준으로 설계한다.

- `Lexical JSON`을 원본 데이터로 저장한다.
- 서버에서 렌더용 `HTML`을 생성한다.
- 목록과 메타 영역에 사용할 `excerpt`를 서버에서 생성한다.

이 구조를 기준으로 두는 이유는 다음과 같다.

- 에디터 상태를 안정적으로 보존할 수 있다.
- 상세 페이지 렌더링과 SEO 대응에 필요한 출력 형태를 확보할 수 있다.
- 이후 이미지 노드나 추가 블록 타입이 생겨도 원본 데이터 모델을 유지하기 쉽다.

### 계층 매핑

`docs/architecture.md` 기준으로 블로그 기능은 다음처럼 배치한다.

- `src/app`: 블로그 관련 route 진입점, metadata, route segment 연결
- `src/views`: 블로그 목록, 상세, 작성, 시리즈 페이지의 페이지 전용 섹션과 조합
- `src/widgets`: 여러 페이지에서 재사용되는 포스트 카드, 댓글 블록, 블로그 셸 같은 공용 조합 UI
- `src/domain`: 블로그 도메인 모델, tRPC procedure, Zod schema, fixture, 재사용 hook, 권한 규칙, 데이터 접근 단위
- `src/core`: Auth.js, Drizzle, Neon 연결, tRPC 초기화, TanStack Query 설정, 환경 설정, 기술 인프라
- `src/shared`: 공용 UI primitive, 포맷터, 범용 유틸

계층 책임은 아래 원칙을 따른다.

- `src/app`은 진입과 연결만 담당한다.
- 페이지 전용 UI는 `src/views`에서 해결한다.
- 여러 페이지에서 반복되는 블로그 조합 UI만 `src/widgets`로 올린다.
- 인증, DB, tRPC, TanStack Query와 같은 기술 기반은 `src/core`에 둔다.
- 블로그 도메인 모델과 도메인별 절차는 `src/domain`에 둔다.

### 도메인 절차 구조 규칙

블로그 관련 `tRPC`는 `src/domain/blog` 아래에서 관리한다. `src/core/trpc`는 기술 기반만 두고, 실제 블로그 procedure와 schema는 도메인 안에 둔다.

권장 구조는 다음과 같다.

```text
src/domain/blog/
  procedure/
    get-list-published/
      index.ts
      schema.ts
      fixture.ts
    get-by-slug/
      index.ts
      schema.ts
      fixture.ts
    post-create-draft/
      index.ts
      schema.ts
      fixture.ts
    post-publish/
      index.ts
      schema.ts
      fixture.ts
    router.ts
  hook/
    use-list-published.ts
    use-create-draft.ts
    use-publish.ts
```

이 구조의 규칙은 다음과 같다.

- `index.ts`: 개별 procedure 구현
- `schema.ts`: Zod 기반 `input`, `output` 스키마 선언
- `fixture.ts`: 해당 스키마를 만족하는 대표 입력/출력 값 정리
- `hook/use-*.ts`: tRPC 기반 `useQuery` 또는 `useMutation` 래퍼
- `procedure/router.ts`: 개별 procedure를 합쳐 `blogRouter`를 만드는 파일

추가 원칙:

- `get-*`는 읽기, `post-*`는 상태 변경 절차로 구분한다.
- 페이지 전용 hook은 `src/views`에 두고, 재사용 가능한 server-state hook만 `src/domain/blog/hook`에 둔다.
- 블로그 외 전역 viewer/auth 성격의 절차가 필요하면 같은 규칙으로 별도 `src/domain/<domain>`을 둘 수 있다.

## 단계별 구현 로드맵

| Phase | 범위 | 완료 기준 |
| --- | --- | --- |
| Phase 0 | 인프라, 인증, 테마, 공용 앱 셸 | Neon, Drizzle, Auth.js, TanStack Query + tRPC, theme 전환이 연결되고 블로그용 전역 레이아웃의 뼈대가 준비된 상태 |
| Phase 1 | 공개 리스트/상세, BOOKLOG 시리즈 읽기 | 비로그인 사용자가 BLOG, MEMO, BOOKLOG 목록과 상세, BOOKLOG 시리즈를 읽을 수 있는 상태 |
| Phase 2 | owner 작성, 초안 저장, 발행 | owner가 전용 작성 페이지에서 글을 작성하고 초안 저장 및 발행할 수 있는 상태 |
| Phase 3 | 좋아요, 댓글, 대댓글 | 로그인 사용자가 좋아요와 댓글, 대댓글을 남길 수 있고 owner가 댓글 삭제를 처리할 수 있는 상태 |
| Phase 4 | SEO, sitemap, 운영 보강 | metadata, sitemap, robots, 에러/빈 상태, 운영 보조 요소가 정리된 상태 |

### Phase 0

- 블로그 기능이 얹힐 인프라와 인증 기반을 먼저 세운다.
- 다크 모드와 라이트 모드를 전역에서 일관되게 다룰 수 있어야 한다.
- 공용 앱 셸은 이후 블로그와 일반 페이지 모두에서 재사용 가능해야 한다.

### Phase 1

- 읽기 경험을 먼저 완성한다.
- BLOG, MEMO, BOOKLOG의 차이는 주로 정보 구조와 목록/상세 표현에서 드러나야 한다.
- 이 단계에서는 쓰기 기능이 없어도 공개 소비 흐름이 완전해야 한다.

### Phase 2

- owner 전용 작성 흐름은 전용 작성 페이지 기준으로 고정한다.
- 초안 저장과 발행을 분리해 운영 안정성을 확보한다.
- `BOOKLOG`는 시리즈와 순서를 함께 입력할 수 있어야 한다.

### Phase 3

- 로그인 사용자 상호작용은 상세 페이지 중심으로 제공한다.
- 상호작용 기능은 읽기 경험을 방해하지 않는 위치와 밀도로 배치한다.
- 권한 체크는 UI뿐 아니라 서버 절차에서도 강제되어야 한다.

### Phase 4

- 기능 구현 후 운영 품질을 보강한다.
- 검색엔진과 공유 미리보기 대응을 포함한다.
- 블로그 제품으로서 필요한 기본 운영 요소를 갖춘다.

## 이번에 하지 않는 것

다음 항목은 이번 문서 기준의 1차 범위에 포함하지 않는다.

- 본문 이미지 업로드
- 첨부파일 업로드
- 태그 시스템
- 검색
- RSS
- 알림
- 다중 관리자 권한

다만 후속 확장을 막지 않도록 다음 전제는 유지한다.

- 에디터 구조는 이미지 블록 추가가 가능한 방향으로 설계한다.
- 데이터 모델은 `cover image`나 본문 미디어 확장이 가능한 형태를 염두에 둔다.
- 권한 체계는 현재 단일 owner 기준이지만, 이후 role 기반 확장이 불가능하지 않도록 고려한다.

## 문서 검토 기준

이 문서는 다음 기준을 만족해야 한다.

- 새로 합류한 개발자가 필요한 페이지 목록과 권한 규칙을 설명할 수 있어야 한다.
- 디자이너가 모바일 우선 화면 목록과 콘텐츠 타입 차이를 이해할 수 있어야 한다.
- 구현자가 각 phase의 범위와 제외 항목을 헷갈리지 않아야 한다.
- `docs/architecture.md`와 충돌하는 계층 책임이 없어야 한다.
- 이후 구현 이슈를 분리할 때 기준 문서로 그대로 사용할 수 있어야 한다.

## 문서 운영 원칙

- 블로그 관련 상위 기준 문서는 현재 이 문서 하나로 유지한다.
- Phase별 상세 구현 계획은 `docs/blog-phases/` 아래 하위 문서로 관리한다.
- 구현 중 새로운 이슈가 생기면 먼저 이 문서의 원칙과 충돌하는지 확인한다.
- 세부 구현 이슈는 이 문서를 요약해서 옮기지 않고, 필요한 문맥만 참조한다.
- 하위 문서가 늘어나더라도 역할 분리는 유지한다.
  - 이 문서는 방향과 경계, 공통 결정을 다룬다.
  - Phase 문서는 구현 순서와 작업 단위를 다룬다.
